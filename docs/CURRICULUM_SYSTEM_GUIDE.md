## 1. System Overview

This is a **personalized, AI-native training platform** that:

- Analyzes a learner's **current skills** (State A) from their profile/resume
- Extracts **target skills** (State B) from a job description or goal
- Computes a **prioritized skill gap** with prerequisites
- Generates a **5-chapter learning path** with deterministic scaffolding + LLM enrichment
- Generates **lesson content on-demand** the moment a learner starts a lesson
- Tracks **skill mastery** per-path and globally (Skill Genome)
- Provides an **AI Mentor** for Socratic coaching
- Grades **implementation tasks** with evidence-based AI assessment
- Detects and recovers from **learner confusion**

### Key Philosophy

- **Deterministic scaffolding, LLM-enriched content**: The path structure (which skills, what order, how many chapters) is deterministic and reproducible. The content (examples, exercises, explanations) is generated by LLMs.
- **On-demand generation**: Lesson content is generated when the learner clicks "Start," not upfront. This avoids wasted compute and allows personalization.
- **AI-native pedagogy**: Learners don't just read about topics — they practice prompting, build artifacts, and get AI-graded feedback.
- **Evidence over submission**: Implementation tasks require proof the output works (screenshots, terminal output, API responses), not just the code itself.

### Tech Stack

| Layer | Technology |
|-------|-----------|
| Backend | Python, FastAPI, SQLAlchemy (async), SQLite/PostgreSQL |
| LLM | Any provider (OpenAI, Anthropic, etc.) via abstraction layer |
| Frontend | React, TypeScript, TanStack Query, Zustand, Tailwind CSS |
| Code Execution | Pyodide (WebAssembly Python in browser) |
| Deployment | Docker Compose |

## 3. Ontology — The Skill Map

The ontology is a **hierarchical skill taxonomy** that powers everything: profiling, gap analysis, path generation, and mastery tracking. It lives in a single JSON file.

### Structure

```json
{
  "version": "2.0.0",
  "proficiency_scale": [
    {"level": 0, "label": "Unaware",      "description": "Has not heard of it"},
    {"level": 1, "label": "Aware",         "description": "Can explain basics"},
    {"level": 2, "label": "User",          "description": "Can apply with help"},
    {"level": 3, "label": "Practitioner",  "description": "Can adapt independently"},
    {"level": 4, "label": "Builder",       "description": "Ships solutions"},
    {"level": 5, "label": "Architect",     "description": "Designs systems"}
  ],
  "layers": [
    {
      "id": "L.FOUNDATION",
      "name": "Foundation",
      "order": 1,
      "domains": ["D.DIG", "D.CTIC"]
    }
  ],
  "domains": [
    {
      "id": "D.PRM",
      "name": "Prompting",
      "layer": "L.APPLICATION",
      "description": "Effective AI interaction techniques",
      "skill_count": 12
    }
  ],
  "skills": [
    {
      "id": "SK.PRM.003",
      "name": "Prompt debugging & iteration",
      "domain": "D.PRM",
      "level": 3,
      "prerequisites": ["SK.PRM.001"],
      "description": "Systematically debug and iterate on prompts"
    }
  ]
}
```

### Hierarchy

```
Layer (8 total)
  └── Domain (22 total)
        └── Skill (186 total, each with prerequisites)
```

### Proficiency Scale (L0–L5)

Every skill assessment, gap, and mastery uses this same 0–5 scale:

| Level | Label | What it means |
|-------|-------|--------------|
| 0 | Unaware | Never encountered |
| 1 | Aware | Can explain the concept |
| 2 | User | Can apply with guidance |
| 3 | Practitioner | Can adapt independently |
| 4 | Builder | Ships production solutions |
| 5 | Architect | Designs systems, teaches others |

### Adapting the Ontology

To use this system for a different domain:

1. Replace the skills JSON with your domain's skill taxonomy
2. Keep the same structure: layers → domains → skills with prerequisites
3. Keep the same proficiency scale (or adapt labels)
4. Skill IDs follow the pattern: `SK.{DOMAIN}.{NUMBER}` (e.g., `SK.PRM.003`)

## 5. Backend Agents

All agents inherit from a `BaseAgent` class that provides:
- `_call_llm(prompt, system_prompt, max_tokens, temperature)` — raw LLM call
- `_call_llm_structured(prompt, output_schema, max_tokens)` — returns validated JSON
- Execution tracking (start/end time, execution ID)

### Agent Inventory

| Agent | Input | Output | Purpose |
|-------|-------|--------|---------|
| **Orchestrator** | Profile + JD | Full analysis | Coordinates all agents |
| **ProfileAnalyzer** | Profile/resume | state_a_skills, top 10 | Assess current skills |
| **JDParser** | Job description | state_b_skills, top 10 | Extract target skills |
| **GapAnalyzer** | State A + B | Prioritized gaps | Rank skill gaps |
| **PathGenerator** | Chapter scaffold | Enriched chapters | Add content to path |
| **ModuleOutline** | Module info | 3-5 lesson outlines | Plan lesson sequence |
| **LessonGenerator** | Module + lesson context | Full lesson content | Generate on-demand content |
| **MentorAgent** | Message + context | Socratic response | Coach the learner |
| **AssessmentAgent** | Skills to test | Quiz questions | Generate/score quizzes |
| **ContentCurator** | Chapter topics | External resources | Find learning resources |

### Orchestrator Workflow

```
Orchestrator.execute()
│
├── [PARALLEL]
│   ├── ProfileAnalyzerAgent → state_a_skills, top_10_current
│   └── JDParserAgent → state_b_skills, top_10_target
│
├── GapAnalyzerAgent → prioritized gaps, summary
│
├── LearningPathGenerator (deterministic) → 5-chapter scaffold
│
├── PathGeneratorAgent (LLM) → enriched chapters
│
└── ContentCuratorAgent → external resources
```

## 6. Deterministic Path Generation

The path scaffold is generated **deterministically** (no LLM) to ensure reproducibility. The LLM only enriches it afterward.

### Algorithm (5 Phases)

#### Phase 0: Expand State A with Prerequisites
- If a learner has Skill X at Level 3, infer they have its prerequisites at Level 2+
- Prevents the path from teaching things the learner already knows

#### Phase 0.5: Enforce Mandatory Domain Coverage
- Every path MUST include skills from these categories:
  - **Foundation** (core literacy)
  - **Applied** (hands-on skills)
  - **Evaluation** (measuring quality)
  - **Safety/Governance** (risk management)
- If any category is missing from the gap, inject the lowest-level skill from that domain

#### Phase 1: Select Top 5 Primary Skills
- Sort gaps by priority
- Pick the 5 highest-priority skills

#### Phase 2: Resolve Prerequisites (Back-Pressure)
- For each primary skill, check if any prerequisites are missing
- If adding prerequisites would exceed 5 chapters, drop the lowest-priority primary skill
- Prerequisites must appear in chapters before the skills that depend on them

#### Phase 3: Enforce Domain Diversity
- No more than 2 chapters from the same domain
- If violated, swap with the next-priority skill from a different domain

#### Phase 4: Build Chapter Scaffold
```json
{
  "chapter_number": 1,
  "skill_id": "SK.PRM.001",
  "skill_name": "Zero-shot prompting",
  "current_level": 1,
  "target_level": 2
}
```

**Critical rule**: `target_level` is always `current_level + 1`. One level up per chapter.

### Why Deterministic?

- **Reproducible**: Same inputs always produce the same path
- **Testable**: You can write unit tests for path generation
- **Debuggable**: No LLM randomness in structural decisions
- The LLM enriches content (examples, exercises) but never changes the structure

## 7. On-Demand Lesson Generation

## 8. AI Mentor (Socratic Coach)

A floating chat panel that coaches learners using the Socratic method — guiding with questions, not giving answers.

### MentorAgent System Prompt

```
You are an AI learning mentor — a warm, supportive, technically precise coach.

YOUR ROLE:
- Help learners understand concepts by asking leading questions (Socratic method)
- Suggest prompts they can try in the Prompt Lab
- Debug code and explain errors clearly
- Encourage AI collaboration — teach them to work WITH AI

RULES:
- NEVER give direct answers to exercises or implementation tasks
- Break the problem down and guide the learner to the solution
- When confused, offer a simpler analogy first
- Keep responses concise (2-4 paragraphs max)
- Adapt language to the learner's skill level

RESPONSE FORMAT:
- Address the learner's question directly
- Provide guidance (not answers)
- End with exactly 2 follow-up prompts on new lines prefixed with "Explore further:"
  - Each must be 25+ words, start with a role instruction ("Act as a...")
  - Must explore DIFFERENT angles from each other
```

### Two Modes

#### Standard Mode
Normal Socratic coaching. Learner asks questions, mentor guides.

#### Implementation Briefing Mode
Triggered when learner clicks "Get AI Mentor Briefing" on an implementation task.
Uses a special system prompt:

```
You are an AI learning mentor preparing a student for an implementation task.

Produce a structured briefing with EXACTLY these 5 sections:

## 1. Assignment Interpretation
What this task is really asking. Core challenge and skills tested.

## 2. What Must Be Built
Bullet list of concrete deliverables (specific artifacts, not vague descriptions).

## 3. Implementation Plan
Numbered step-by-step checklist (5-10 steps, action verbs).

## 4. Tools Likely Required
Bullet list of tools with what each is used for.

## 5. Required Artifacts
Bullet list of files/outputs with expected formats.

Keep under 800 words. Be practical, not theoretical.
```

### Workspace Handoff

## 10. Implementation Tasks & Grading

### 3-Step Workflow

```
Step 1: Review Assignment (auto-completed — learner is reading the card)
Step 2: Get AI Mentor Briefing (click "Ask AI Mentor")
Step 3: Submit & Get Graded (upload files → AI grades)
```

### Upload Slots

Upload slots are built from `evidence_requirements` in the lesson content:

```json
"evidence_requirements": [
  {"name": "Working Script", "description": "Python file with solution", "format": "code"},
  {"name": "Execution Proof", "description": "Screenshot showing correct output", "format": "screenshot"}
]
```

Each slot requires at least one file. Accepted types: PDF, DOCX, code files, images (max 10MB each).

### Grading System Prompt

```
You are an expert AI learning assessor grading a learner's implementation task submission.

Evaluate and return a JSON object with:
{
  "score": <integer 0-100>,
  "passed": <boolean, true if score >= 70>,
  "feedback": "<2-3 sentence overall assessment>",
  "strengths": ["<strength 1>", "<strength 2>"],
  "improvements": ["<improvement 1>", "<improvement 2>"]
}

Grading criteria:
- Does the submission address ALL stated requirements? (40 points)
- Quality and completeness of the deliverable (30 points)
- Evidence of understanding (not just copy-paste) (20 points)
- Presentation and clarity (10 points)

Score ranges:
- 0-20: Empty or clearly unrelated
- 40-69: Partially addresses requirements
- 70-90: Fully addresses requirements with good quality
- 90+: Exceptional work that goes beyond requirements
```

### Grading Prompt Template

```
TASK: {title}
DESCRIPTION: {description}
DELIVERABLE: {deliverable}
REQUIREMENTS:
1. {requirement 1}
2. {requirement 2}

LEARNER'S SUBMISSION (Attempt #{attempt_number}):
{extracted content from uploaded files}

Grade this submission against the requirements above.
```

### File Processing Pipeline

1. Accept uploaded files (PDF, DOCX, code, images)
2. Extract text from each file (PDF parsing, DOCX parsing, code as-is, OCR for images)
3. Combine all extracted text
4. Truncate to 12,000 characters (LLM context limit)
5. Send to LLM with grading prompt
6. Parse JSON response
7. Save submission record with attempt tracking
8. Update Skill Genome if passed

### Pass Threshold

Score >= **70** = passed. Learner can resubmit unlimited times.

### Simulation Mode

A "Simulate Submission (Demo)" button generates realistic deliverable files and returns a canned passing grade (score=85). Useful for demos and testing.

## 11. Skill Genome (Global Mastery Tracking)

The Skill Genome tracks mastery **across all learning paths** — it's the learner's permanent skill record.

### How It's Updated

Uses **Exponential Moving Average (EMA)** to blend new evidence with existing mastery:

```python
EMA_WEIGHT = 0.7
new_mastery = EMA_WEIGHT * current_mastery + (1 - EMA_WEIGHT) * new_signal
# Keeps 70% of existing assessment, blends in 30% of new evidence
```

### Evidence Sources

| Source | Signal Calculation |
|--------|-------------------|
| Quiz (knowledge check) | `quiz_score / 100 * target_level` |
| Lesson completion | `(current_level + target_level) / 2` |
| Project (implementation task) | `feedback_quality * target_level` |
| Mentor interaction | `0.5 * target_level` (low-weight signal) |

### Confidence

```python
confidence = min(1.0, evidence_count / 10)
# 1 evidence = 10% confidence
# 5 evidence = 50% confidence
# 10+ evidence = 100% confidence
```

### SkillMastery vs. SkillGenome

| Feature | SkillMastery | SkillGenome |
|---------|-------------|-------------|
| Scope | Per learning path | Global (all paths) |
| Updates | On lesson completion | On any evidence signal |
| Level range | initial → target | 0.0 → 5.0 |
| Purpose | Path progress tracking | Permanent skill record |

## 17. Design Patterns & Principles

### 1. Deterministic Scaffold + LLM Enrichment
The **structure** (which skills, what order, how many chapters) is deterministic and reproducible. The **content** (examples, exercises, explanations) is generated by LLMs. This gives you the best of both worlds: testable structure + personalized content.

### 2. On-Demand Generation
Don't generate everything upfront. Generate lesson content when the learner starts it. Cache in the database. Idempotent endpoints mean re-requests return the same cached content.

### 3. Evidence-Based Assessment
Don't just ask for code — ask for code AND proof it works. Screenshots, terminal output, API responses. The grading LLM evaluates the complete submission.

### 4. EMA Mastery Tracking
Use Exponential Moving Average for skill tracking. New evidence blends with existing assessment (70/30 split). Prevents a single bad quiz from tanking overall mastery.

### 5. Socratic Mentoring
The AI Mentor never gives direct answers. It asks leading questions, offers analogies, and suggests prompts to try. This builds understanding, not dependence.

### 6. Event-Driven Component Communication
Components communicate via DOM custom events (`open-mentor`, `mentor-responded`). This keeps components decoupled — the implementation task card doesn't import MentorChat.

### 7. Confusion as a Signal
Confusion isn't a failure — it's data. When learners signal confusion, the system generates targeted recovery content and logs the event for analytics.

### 8. Multi-Agent Orchestration
Each agent has one clear responsibility and structured JSON output. The orchestrator coordinates them, running independent agents in parallel.

### 9. Completion Gating
Lessons cannot be marked complete without answering knowledge checks and passing the implementation task. This ensures learners actually engage with the material.

### 10. Mandatory Domain Coverage
Every learning path includes foundation, applied, evaluation, and safety/governance skills — even if the job description doesn't explicitly mention them.
